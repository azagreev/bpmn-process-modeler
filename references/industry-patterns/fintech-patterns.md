# Onboarding & Identity

## 1. KYC (retail client)

Обязательные этапы в процессе:

1. Сбор персональных данных (ФИО, паспорт / паспорт иностранца, адрес регистрации и фактический, источник дохода / средств, контакты)
2. Биометрия — фото документа + селфи (liveness check)
3. Верификация документов — Service Task с интеграцией в KYC-провайдера (Sumsub, Regula, IDnow) или локальное бюро (ЕСИА в РФ, UzID в UZ, eGov в KZ, E-Gov в BY)
4. AML-скрининг — параллельный шлюз на три проверки: санкционные списки, PEP, adverse media
5. Risk scoring — Business Rule Task (DMN) либо external provider
6. Решение — XOR gateway: одобрить / дополнительная проверка (Enhanced Due Diligence) / отклонить
7. Уведомление клиента + запись в реестр клиентов

**Compliance-аннотации (обязательно на диаграмме):**
- РФ: 115-ФЗ «О противодействии легализации», ст. 7 (идентификация до приёма средств). Внутренний скрининг — перечень Росфинмониторинга.
- Международный скрининг (OFAC / EU / UN / HMT) — при трансграничных операциях и корреспондентских отношениях
- FATF 40 Recommendations — базовая рамка для всех юрисдикций
- ЕАЭС: локальные аналоги (Закон UZ «О противодействии легализации», Закон KZ № 191-IV, Закон BY № 165-З)
- Срок хранения KYC-данных — минимум 5 лет после окончания отношений с клиентом
- re-KYC триггеры — отдельная аннотация с пометкой «⚠ Уточнить пороги по юрисдикции» (ст. 6 115-ФЗ для РФ: обязательный контроль при 600 000 ₽; для UZ/KZ/BY/TR — локальные пороги требуют сверки перед имплементацией)

**Особенности моделирования:**
- AML-скрининг моделируется как параллельный шлюз (AND split/join) — три проверки идут одновременно, результат агрегируется
- На шаге решения при результате «Enhanced Due Diligence» — петля возврата на шаг 1 с расширенным списком документов (approval loop pattern, см. `bpmn-patterns.md #1`)
- Для wallet-онбординга добавляется шаг привязки номера телефона и выпуска wallet ID (Service Task с отправкой SMS и Timer boundary на подтверждение)

## 2. KYB

Юридическое лицо, ИП, самозанятый. Применяется для открытия расчётного счёта, merchant-аккаунта, выдачи кредита, BaaS-интеграции.

1. Сбор учредительных документов (Устав, решение о создании, ЕГРЮЛ/ЕГРИП выписка, ОГРН/ИНН, лицензии для регулируемых видов деятельности)
2. Проверка документов — Service Task с интеграцией в ФНС (открытые данные) / UZ Тasqilot / KZ Egov / локальные регистры
3. UBO / beneficial ownership — отдельный sub-process: раскрытие бенефициаров до 25% + KYC каждого
4. Проверка деловой репутации — открытые судебные базы, банкротный реестр, неплательщик налогов
5. AML для компании — санкции, PEP-связи бенефициаров, adverse media
6. Risk rating + лимиты по счёту / merchant-аккаунту — Business Rule Task (DMN)
7. Решение — XOR gateway: одобрить / EDD / отклонить
8. Подписание договора — User Task + e-sign (УКЭП в РФ, eIDAS в EU, локальные аналоги)

**Compliance-аннотации:**
- 115-ФЗ, ст. 7.1 (идентификация представителя и бенефициара)
- Положение ЦБ РФ 499-П (идентификация клиентов — юр.лиц)
- Для самозанятых (НПД): проверка статуса через API ФНС «Мой налог»
- Срок хранения документов — 5 лет после прекращения договорных отношений
- Периодический пересмотр (re-KYB) — не реже 1 раза в год для клиентов высокого риска, раз в 3 года для низкого

**Особенности моделирования:**
- UBO-sub-process часто выносится в отдельную диаграмму (level 1) — там внутри цикл по каждому бенефициару с индивидуальной проверкой
- Для merchant-аккаунта добавляется проверка ОКВЭД на соответствие разрешённым категориям (исключения по 115-ФЗ: казино, оружие, дистанционная торговля лекарствами в РФ и т.д.)

## 3. Seller / Merchant KYB

Вариант KYB для онбординга селлера на маркетплейс. Отличается от классического KYB тем, что у селлера есть техническая интеграция (API / витрина) и операционные лимиты по обороту.

1. Регистрация селлера на портале маркетплейса — User Task (селлер через self-service)
2. Базовый KYB (см. процесс 2) — Service Task
3. Проверка соответствия профиля магазина и ОКВЭД — Business Rule Task
4. Подписание акцепта оферты маркетплейса — User Task + e-sign
5. Параллельный шлюз:
   - Выпуск merchant-аккаунта в платёжной системе (card / wallet / payout) — Service Task
   - Выдача API-ключей для витрины / 1С / remote storage integration — Service Task
   - Создание аккаунта в системе логистики маркетплейса — Service Task
6. Тестовый заказ (E2E-проверка integration + payout) — Sub-process
7. Активация селлера — End Event «Селлер активен»

**Compliance-аннотации:**
- 54-ФЗ в РФ (онлайн-касса для приёма платежей в пользу селлера)
- Для транспарентности в РФ: агентский договор + чек маркетплейса в пользу селлера
- Для UZ/KZ/TR — локальные требования к агентской модели и НДС
- Запрет на онбординг запрещённых ОКВЭД: оружие, табак онлайн, алкоголь в РФ, криптовалюты

**Особенности моделирования:**
- Обязательно collaboration pool: «Селлер» + «Маркетплейс» + «Платёжная система» + «Логистика» с message flows между пулами
- Тестовый заказ — это отдельная диаграмма с петлёй «не прошло → retry» (approval loop)
- SLA на активацию обычно 1–3 рабочих дня — аннотация Timer Boundary на шаге 6

## 4. BaaS partner onboarding

Для случая, когда маркетплейс / финтех интегрируется с банком-партнёром по модели Banking-as-a-Service (примеры: n11 ↔ Fibabanka, wildberries wallet, иностранные SPV ↔ local bank).

1. Initial scoping — User Task (бизнес-встреча, MoU)
2. Regulatory due diligence банка — Sub-process: проверка лицензий, рейтинга, санкционной чистоты; проверка собственной юрисдикции на разрешения для BaaS-модели
3. Параллельный шлюз:
   - Юридический due diligence — анализ договора BaaS, ограничений по комиссиям, liability sharing, SLA
   - Технический due diligence — API-контракты, throughput, uptime, security, PII handling
   - Комплаенс-alignment — AML-политика, санкционные списки, отчётность
4. Подписание Master Services Agreement — User Task (multi-party)
5. Sandbox-интеграция и сертификация — Sub-process: тестовые сценарии (KYC, payment auth, refund, chargeback, payout, reporting)
6. Production cut-over — Service Task с compensation (rollback)
7. Go-live + incident response protocol — End Event «Live»

**Compliance-аннотации:**
- Для Турции: BDDK Yönetmelik на сервисный банкинг (Servis Bankacılığı), требования к captive-моделям
- Для РФ: Положение 266-П о банковских картах, требования к агентам по приёму платежей
- Data residency: где хранятся персональные данные клиентов (152-ФЗ в РФ запрещает первичное хранение ПДн граждан РФ за рубежом)
- Liability sharing при мошенничестве, ошибках, регуляторных штрафах — отдельно на диаграмме как аннотация на договорном этапе

**Особенности моделирования:**
- Обязательно collaboration pool: «Маркетплейс» + «Банк-партнёр» + «Регулятор» + иногда «Технический вендор» (интегратор, VASP)
- Message flows между всеми парами участников
- SLA: BaaS-онбординг обычно 3–9 месяцев — Timer Boundary с эскалацией при просрочке
- Sandbox sub-process — отдельная диаграмма; часто повторяется несколько раз до получения PASS на сертификационные сценарии

# Payments & Transfers

## 5. Payment authorization (card / wallet / СБП)

1. Receive Task — запрос от мерчанта / партнёра (message flow, если collaboration)
2. Параллельный шлюз:
   - Fraud check — Service Task или Business Rule Task (DMN)
   - Velocity check — Business Rule Task (количество транзакций в окне)
3. Проверка баланса / кредитного лимита / доступных средств — Service Task
4. Routing — XOR по типу транзакции: внутренний wallet / СБП / локальная карточная схема / cross-border
5. Authorization request к issuing bank (если карта) — Service Task
6. Response mapping — Script Task
7. Send Task — ответ мерчанту + receipt клиенту

**Compliance-аннотации:**
- Таймауты end-to-end: Мир / Visa / Mastercard — 3–8 сек; СБП — 3 сек (Положение ЦБ РФ 732-П)
- SCA по 161-ФЗ (РФ) / PSD2 (EU) / BDDK (TR) — обязательно для платежей выше локальных порогов
- PCI DSS при обработке номеров карт; если используется tokenization — уровень снижается до SAQ-A

**Особенности моделирования:**
- Fraud и velocity — параллельно, агрегация в inclusive gateway (обе должны пройти)
- При отказе fraud — сразу End Event «Отказ по фрод-правилам», без повторной попытки авторизации
- Если требуется SCA — переход в sub-process «3DS / SCA challenge» (см. процесс 6)
- Internal service tasks (delegate pattern в C7) часто используются для fraud и routing; external workers — для вызовов issuing bank и СБП

**Платёжные коридоры в РФ (аннотация к шагу Routing):**
- Внутри РФ: Мир, СБП, международные схемы через локальных эквайеров (частично)
- Для трансграничных: разрешённые коридоры через ЕАЭС (национальные платёжные системы), CIPS (Китай), SPFS (ограниченно), remaining correspondent banking к «белым» юрисдикциям
- SWIFT — частично ограничен, нельзя предполагать доступность для всех пар стран

## 6. 3DS / SCA challenge

Короткий sub-process, вызывается из Payment authorization когда требуется усиленная аутентификация.

1. Issuer decides Challenge vs Frictionless — XOR gateway (в 3DS 2.x на стороне ACS)
2. Frictionless — End Event «SCA passed» (риск-анализ достаточен)
3. Challenge — Send Task: редирект клиента на страницу банка / push в мобильное приложение
4. User Task — клиент вводит код / подтверждает в приложении
5. Timer Boundary — 5 минут; при истечении → End Event «SCA timeout»
6. Receive Task — ACS возвращает результат
7. XOR — успех / отказ

Референс для BPMN-паттерна: см. `bpmn-patterns.md` (event-based gateway на шаге 4-5 для event «пользователь подтвердил» или event «таймаут»).

**Compliance:** динамический код обязателен для transaction-binding (161-ФЗ ст. 8, PSD2 RTS Art. 5). Exemption для low-value (до 30 EUR / 3000 ₽) — можно без challenge.

## 7. P2P / Cross-border transfers

1. Message Start Event — запрос на перевод (СБП / wallet-to-wallet / трансграничный)
2. Sender KYC/AML check — Business Rule Task (обычно уже пройден, но re-check на санкционные флаги)
3. Recipient validation — Service Task: проверка номера / кошелька / IBAN / phone
4. Sanctions screening обеих сторон — Service Task (даже при локальных переводах)
5. Limit check — Business Rule Task: дневной / месячный лимит sender + receiver
6. Debit sender + credit recipient — два Service Task внутри XA-транзакции либо через compensation pattern (см. `bpmn-patterns.md`)
7. Уведомление обеим сторонам + receipt — два Send Task

**Compliance-аннотации:**
- 115-ФЗ, ст. 7.2 — идентификация при переводах без открытия счёта свыше 15 000 ₽
- Положение ЦБ РФ 383-П (правила осуществления переводов)
- ЕАЭС: Решение Совета ЕЭК № 12 по общим принципам платёжных систем

**Корридорная таблица (аннотация на Routing или отдельный XOR gateway):**

| Коридор | Типичный канал | Ограничения |
|---|---|---|
| RU → UZ (wallet / P2P) | ВТБ, Газпромбанк ↔ Узнацбанк, Capital Bank; национальные системы HUMO / Uzcard; Золотая Корона; Alipay+ через HUMO | Валюта: рубль → сум, курс ЦБ UZ; лимиты 115-ФЗ |
| UZ → RU | HUMO / Uzcard ↔ Мир; трансграничные переводы через партнёрские банки | Ограничения по ЕАЭС-классификации; требования CBU Resolution 3618 |
| RU → KZ | Мир ↔ Kaspi / Halyk; прямые корр.отношения ВТБ ↔ Halyk | ARDFM регулирование; pаскрытие источника для сумм свыше ~1.5 млн тенге |
| KZ → RU | Kaspi Gold, Halyk, Forte Bank ↔ российские банки через Мир | Аналогичные ARDFM требования |
| RU → TR | Разрешённые альтернативные каналы (zolotaya korona через local partners, стейблкоины — серая зона) | Санкционная чувствительность к cross-border; BDDK требования к beneficiary bank |
| TR → RU | Ограничено — преимущественно через наличные / crypto off-ramps | Для wallet — только через партнёрские банки с compliance clearance |
| RU → ET | Через альтернативные каналы, crypto off-ramps, реже — через белые юрисдикции (UAE, Hong Kong) | Effectively no SWIFT для розничных P2P |

Аннотации к этой таблице — с пометкой «⚠ Уточнить актуальный канал на дату моделирования» — реальность быстро меняется.

**Особенности моделирования:**
- Всегда collaboration pool из минимум двух банков (sender bank + receiver bank) + иногда clearing / settlement system
- Compensation критична на шаге 6: debit прошёл, credit упал — автоматический rollback debit
- Timer Boundary на всём процессе (обычно 30–60 сек для instant payment) — при истечении escalation на manual review

## 8. Chargeback / dispute

1. Message Start Event — клиент оспаривает операцию (в банке-эмитенте)
2. Issuer логирует chargeback + блокирует сумму на merchant-аккаунте — Service Task
3. Issuer отправляет chargeback notification в acquirer — Send Task (через card scheme)
4. Acquirer → merchant (маркетплейс) — Message Catch + Send Task
5. Merchant предоставляет доказательства (представление товара / оказания услуги) — User Task + Timer Boundary (обычно 10–45 дней в зависимости от schema + reason code)
6. Issuer оценивает доказательства — User Task
7. XOR — principled в пользу клиента (возврат средств) vs в пользу merchant (разблокировка) vs arbitration (вторая итерация)
8. End Event в зависимости от исхода

Референс для BPMN-паттерна: collaboration pool с 4+ участниками (Клиент, Issuer, Acquirer, Merchant), message flows между пулами.

**Compliance:**
- Правила card schemes (Visa, Mastercard, Мир) — reason codes, сроки, обязательные документы
- 161-ФЗ ст. 9 — право клиента на оспаривание в течение 30 дней для РФ
- Регламенты СБП — отдельный chargeback-процесс с более короткими сроками

---

## 9. Refund

1. Message Start Event — инициация возврата (merchant или клиент через support)
2. Проверка возможности возврата — Business Rule Task (XOR: full refund / partial / отклонено)
3. Списание с merchant-счёта — Service Task
4. Возврат на источник оплаты — XOR: карта (reverse authorization) / wallet / BNPL (корректировка графика) / иной
5. End Event «Возврат выполнен»

Короткий процесс. Для возврата товара на маркетплейсе — читать параллельно с `marketplace-patterns.md` #Returns: там товарная часть, здесь денежная.

**Аннотации:** SLA на возврат на карту — до 30 дней (Положение 266-П), обычно 3–10 дней; для BNPL — пересчёт графика в течение 1–3 рабочих дней.

# Lending

## 10. BNPL / Installment approval

1. Message Start Event — заявка клиента
2. KYC (если не пройден ранее) — см. процесс 1
3. Bureau check — Service Task: интеграция в кредитное бюро (НБКИ / ОКБ / Эквифакс в РФ; UzKredBureau; KZ Credit Bureau; Findeks в TR)
4. Affordability + DTI check — Business Rule Task (DMN): отношение платежа к доходу, свободный денежный поток
5. Offer generation + ПСК / полная стоимость кредита — Script Task
6. User Task — клиент принимает / отклоняет оффер; Timer Boundary 15–30 мин
7. Первый платёж / авторизация карты — Service Task с compensation (refund + отмена графика)
8. Формирование графика платежей + первая дата списания — Service Task
9. End Event «BNPL активирован»

**Compliance-аннотации:**
- 353-ФЗ «О потребительском кредите» ст. 6 — раскрытие ПСК в рамке на первой странице оффера
- 353-ФЗ ст. 5 — индивидуальные условия договора (обязательная таблица)
- Лицензия: МФО (для микрозайма до 1 млн ₽) / банк / KBО (для BNPL без выдачи наличных) — зависит от юридической конструкции
- Для Турции: BDDK regulations на taksit ticari (коммерческие платежи в рассрочку), отдельно от потребкредита
- Для UZ/KZ: локальные законы о потребительском кредитовании + требования по бюро

**Особенности моделирования:**
- Шаги 3–4 часто параллельные (bureau + affordability) для ускорения решения
- На шаге 7 обязательна компенсация: если первый платёж упал — график не создаётся, клиент получает сообщение
- Для repeat клиентов (уже проходивших KYC) шаг 2 пропускается — XOR после Start Event «первая сделка?»

## 11. Collection

Sub-process BNPL или потребкредита, вызывается при пропуске планового платежа.

1. Timer Start Event — плановая дата списания
2. Attempt списания — Service Task
3. XOR — успех → End; неудача → продолжение
4. Мягкое напоминание (SMS / push) — Send Task + Timer Boundary 1–3 дня
5. Повторная попытка списания
6. XOR — по-прежнему неудача → продолжение
7. Жёсткое уведомление + предложение реструктуризации — User Task (клиент может принять новый график)
8. Эскалация в внутренний collection-отдел / передача в агентство — Send Task
9. Legal action — sub-process (судебный порядок, 353-ФЗ ст. 12-14)

**Аннотации:** 230-ФЗ «О защите прав при возврате просроченной задолженности» — ограничения на частоту, время, способы коммуникации с должником. Для РФ — обязательно.

## 12. Leasing

1. Message Start Event — заявка от лизингополучателя
2. KYB лизингополучателя — см. процесс 2
3. Выбор предмета лизинга — User Task (лизингополучатель или совместно с лизинговой компанией)
4. Оценка предмета + проверка продавца — Service Task
5. XOR по типу лизинга: финансовый / операционный / возвратный (sale-leaseback) — определяет структуру договора
6. Credit decision + scoring — Business Rule Task (внутренняя модель лизинговой компании)
7. Одобрение + график платежей — User Task
8. Подписание договора лизинга — User Task + e-sign
9. Service Task: покупка предмета у продавца + постановка на учёт
10. Передача предмета лизингополучателю — Send Task / Manual Task
11. Sub-process: мониторинг (ежемесячно графики платежей, ежеквартально техсостояние для дорогих предметов)

**Compliance-аннотации:**
- 164-ФЗ «О финансовой аренде (лизинге)» — базовый закон в РФ
- НДС вычет для лизингополучателя — важно для бухгалтерии (не BPMN-контекст, но частая аннотация)
- Для автотранспорта / недвижимости: 39-ФЗ об инвестиционной деятельности + регистрационные действия
- Субсидии и программы господдержки (Минсельхоз, Минпромторг) — отдельный Business Rule Task

**Особенности моделирования:**
- Тип лизинга — XOR gateway на раннем этапе; от выбора зависит структура договора и налоговый учёт
- Для возвратного лизинга (sale-leaseback) — добавляется обратный sub-process «продажа актива лизинговой компании» перед передачей в лизинг

## 13. Factoring

1. Message Start Event — поставщик (фактор-клиент) передаёт дебиторскую задолженность
2. Верификация инвойсов — Service Task: проверка счёта-фактуры, УПД, подписей, первичных документов
3. Параллельный шлюз:
   - Оценка риска поставщика — Business Rule Task
   - Оценка риска дебитора — Business Rule Task (часто это крупные компании, у них свои bureau-данные)
   - Проверка факта поставки — Service Task (акт приёма-передачи, ТТН)
4. Решение — XOR: финансировать % от инвойса / отклонить / запросить доп.документы
5. Выплата аванса поставщику (обычно 70–90% от инвойса) — Service Task
6. Ожидание оплаты дебитором — Receive Task + Timer Boundary (срок оплаты по договору + регрессное окно)
7. Дебитор оплачивает фактору — Message Catch Event
8. Расчёт комиссии + остаток поставщику — Script Task + Service Task
9. End Event «Факторинговая сделка закрыта»

**Compliance-аннотации:**
- 395-1-ФЗ «О банках и банковской деятельности» — факторинг как банковская операция
- 824 ГК РФ «Финансирование под уступку денежного требования» — правовая база
- Для международного факторинга — правила GRIF (General Rules for International Factoring)
- Regress vs non-regress модель — ключевое различие в риск-аллокации (аннотация на ранний gateway)

**Особенности моделирования:**
- Тип факторинга — XOR в начале: с правом регресса / без права регресса / reverse factoring (supply chain finance, инициирует дебитор)
- Таймер на шаге 6 критичен: если дебитор не платит в срок + не-регресс — убыток фактора; с регрессом — списание с поставщика

## 16. Seller lending (marketplace)

Кредитование активного селлера маркетплейса (ЮЛ / ИП / самозанятый) со стороны кэптивной МКК или банка-партнёра. Ключевое отличие от BNPL для физлиц (процесс 10) — это B2B-заём, 353-ФЗ не применяется; основной канал погашения — приоритетное автоматическое удержание из выручки селлера в пользу маркетплейса на основании зачёта встречных однородных требований (ст. 410 ГК РФ).

1. Message Start Event — заявка селлера из ЛК маркетплейса / портала МКК (выбор суммы + срока)
2. XOR gateway «первая заявка / повторная» — для повторных KYB сокращён до re-check санкций и статуса селлер-аккаунта
3. KYB селлера (для первой заявки) — см. процесс 3 «Seller / Merchant KYB»
4. AND Gateway (параллельный fetch) → Business Rule Task (DMN): оборот и динамика продаж из витрины маркетплейса, категория товаров, доля возвратов / жалоб, данные БКИ (НБКИ / ОКБ / Эквифакс), санкционный скрининг, DSCR / коэффициент покрытия долга — агрегация в скоринговое решение; SLA 1–3 рабочих дня (⚠ Уточнить для автоматической / полуавтоматической ветки)
5. XOR gateway — одобрить / отклонить / EDD; при EDD — User Task запроса доп.документов с петлёй возврата на шаг 4
6. Script Task — формирование оффера: сумма, срок, ставка, график, комиссия (⚠ Уточнить диапазоны в продуктовом каталоге)
7. User Task — селлер акцептует оффер УКЭП ЮЛ / ИП по 63-ФЗ; Timer Boundary (interrupting) 7–14 дней (⚠ Уточнить в продуктовом каталоге); при истечении — End Event «Оффер не акцептован»
8. Service Task «Disbursement» — перевод на расчётный счёт селлера через расчётный банк, SLA 1–3 рабочих дня; **compensation boundary event** — при выявлении fraud-flag / ошибки KYB / отзыва согласия после перевода срабатывает compensation handler (reverse-платёж, аннулирование договора, снятие приоритетного удержания); см. compensation pattern в `bpmn-patterns.md`
9. Service Task — формирование графика платежей + первая плановая дата; активация приоритетного удержания в биллинге маркетплейса на основании оговорки о зачёте по ст. 410 ГК РФ
10. Timer Start Event (ежедневно / еженедельно в зависимости от графика, cron-подобное выражение) — запуск цикла «Попытка списания»
11. XOR gateway по каналу списания — (a) зачёт из выручки маркетплейса (приоритет) → Service Task зачёта; (b) прямое списание с р/с по заранее данному акцепту → Service Task через расчётный банк; (c) ручное пополнение реквизитов МКК → Receive Task (Intermediate Message Catch)
12. XOR gateway «списание успешно / не успешно» — успех: Script Task уменьшения остатка долга → End Event «Плановый платёж проведён»; неуспех: эскалация в sub-process «Seller collection» (см. будущий процесс — адаптация процесса 11 для B2B)
13. End Event «Заём погашен полностью» — при обнулении остатка долга; параллельно Service Task снятия приоритетного удержания и закрытия сделки

**Compliance-аннотации:**
- **151-ФЗ «О микрофинансовой деятельности»** — если кредитор МКК; максимальная сумма предпринимательского займа МКК ограничена (⚠ Уточнить действующий лимит по реестру ЦБ РФ и редакции 151-ФЗ на дату моделирования)
- **395-1-ФЗ + Положение ЦБ РФ 590-П** — если кредитор банк; формирование резервов на возможные потери по корпоративным ссудам (аннотация на шаге 8)
- **115-ФЗ, ст. 7.1 + Положение ЦБ РФ 499-П** — идентификация юрлица / ИП, представителей и бенефициаров (шаг 3 KYB)
- **63-ФЗ «Об электронной подписи»** — УКЭП ЮЛ или ИП на шаге 7; Госуслуги / ЕСИА для ЮЛ **не применяются**; для ИП допустима УКЭП ФЛ с отражением статуса ИП в сертификате
- **Ст. 410 ГК РФ** — зачёт встречных однородных требований как правовое основание автосписания из выручки маркетплейса; условие о зачёте должно содержаться и в оферте маркетплейса, и в договоре займа
- **54-ФЗ** — кассовый учёт при взаимозачёте (позиции ФНС и методологов расходятся: часть требует чек на зачёт, часть считает его не операцией расчётов; ⚠ Уточнить актуальную позицию ФНС и внутренней бухгалтерии)
- **230-ФЗ «О защите прав при возврате просроченной задолженности»** — при просрочке → sub-process Seller collection; ограничения по частоте, времени суток и каналам коммуникации с должником-ЮЛ применяются в адаптированном виде (⚠ Уточнить действие 230-ФЗ на B2B-должников в текущей редакции)
- **НК РФ ст. 269** — контролируемая задолженность и тонкая капитализация; аннотация обязательна, если кредитор и маркетплейс входят в одну группу лиц
- **НК РФ ст. 251, 265** — налоговый учёт процентов у заёмщика-ЮЛ на ОСН; для ИП и ЮЛ на УСН / ПСН — отдельный порядок признания
- **Положение ЦБ РФ 4212-У** — отчётность МКК перед ЦБ РФ; сопрягается с процессом 15 Regulatory reporting
- Срок хранения документов по сделке — минимум 5 лет после прекращения отношений с клиентом (115-ФЗ ст. 7 п. 4); срок хранения договора займа — в рамках сроков исковой давности по ГК РФ
- 353-ФЗ, ПСК, таблица индивидуальных условий — **не применяются** (норма потребкредита для физлиц)

**Особенности моделирования:**
- Collaboration pool обязателен, минимум 4 пула: **Селлер** (подача заявки, акцепт оффера, ручное погашение), **Маркетплейс** (витрина, данные продаж для скоринга, биллинг с зачётом), **МКК / банк-кредитор** (underwriting, оффер, disbursement, compensation, графики, collection), **Расчётный банк селлера** (зачисление disbursement, списание по акцепту)
- Message flows между пулами, минимум 6: (1) Селлер → Маркетплейс «заявка» и в обратную сторону «оффер»; (2) Маркетплейс → МКК «поток данных о продажах, возвратах и категориях для скоринга»; (3) МКК → Расчётный банк «платёжное поручение на disbursement»; (4) Маркетплейс → МКК «сумма зачёта из выручки за период»; (5) МКК → Расчётный банк «поручение на прямое списание по акцепту»; (6) Расчётный банк → МКК «acknowledgement зачисления / списания»
- Шаги 1, 4, 6–9, 11 (ветка a), 13 — happy path на центральной линии диаграммы слева направо (согласно рекомендациям Camunda по emphasizing the happy path); альтернативные ветки (EDD, отказ, истечение оффера, compensation, неуспех списания) — выше или ниже центра
- Compensation boundary event на шаге 8 — handler с атрибутом `isForCompensation="true"` выполняет последовательно: reverse-перевод, аннулирование договора и графика, снятие приоритетного удержания; триггер — Compensation Throw Event из ветки обнаружения fraud-flag или ошибки KYB
- Шаг 10 — Timer Start Event отдельного процесса «Попытка списания» с cycle-выражением по расписанию графика; один экземпляр на одного активного селлера за цикл
- Повторные заявки (XOR на шаге 2) идут по сокращённой ветке: пропуск шага 3 KYB, ускоренный underwriting на шаге 4 с приоритетом исторических данных из внутренних систем маркетплейса
- Sub-process'ы, не раскрываемые в этой версии — только ссылки: (1) **Seller collection** → см. будущий процесс (адаптация процесса 11 для B2B с учётом 230-ФЗ и механики блокировки вывода средств / принудительной распродажи остатков); (2) **Реструктуризация графика** → см. будущий процесс; (3) **Pre-disbursement verification** → см. будущий процесс (проверка корректности реквизитов, fresh sanctions pull, PEP-match бенефициара на момент выдачи)
- Не моделировать retry-поведение вызовов внешних API и попыток списания внутри диаграммы — это уровень Camunda runtime (job workers, incident handling), не уровень BPMN-модели
- Именование элементов — object + verb в инфинитиве для задач, object + state для событий, вопрос для XOR-gateway с ответами на исходящих потоках

## 17. Revenue-Based Financing (RBF) / Merchant Cash Advance

Финансирование активного селлера маркетплейса, где погашение осуществляется через приоритетное удержание фиксированного % от каждого payout-цикла выручки селлера в пользу кредитора — МКК или банка-партнёра. В отличие от BNPL (#10) нет жёсткого графика платежей и процентной ставки в классическом виде; в отличие от Seller lending (#16) нет Timer Start Event на плановое списание — триггером удержания выступает каждое поступление выручки. End Event — по достижению накопленной суммы repayment cap, не по дате. Юридическая конструкция в РФ строится как гибрид «заём + оговорка о зачёте встречных однородных требований по ст. 410 ГК РФ»; чистый MCA с factor rate вместо % не вписывается в 151-ФЗ и 395-1-ФЗ и требует перевода удорожания в % per annum + единовременную комиссию.

1. Message Start Event — заявка селлера из ЛК маркетплейса / портала кредитора (выбор суммы + желаемого holdback %)
2. XOR gateway «первая заявка / повторная» — для повторных KYB сокращён до re-check санкций и статуса селлер-аккаунта; XOR gateway «есть ли активное обязательство по действующему RBF» → если да, ветка отказа / оффера только после погашения ≥50–75% текущего
3. KYB селлера (для первой заявки) — см. процесс 3 «Seller / Merchant KYB»
4. AND Gateway (параллельный fetch) → Business Rule Task (DMN): оборот и волатильность продаж из витрины маркетплейса, категория товаров, доля возвратов/жалоб, доля off-platform активности, DSCR / покрытие долга по historical cash flow, санкционный скрининг, БКИ (НБКИ / ОКБ / Эквифакс в РФ; Findeks в TR; First Credit Bureau в KZ) — агрегация в скоринговое решение; SLA 1–3 рабочих дня, для повторных ≤24 часов
5. XOR gateway — одобрить / отклонить / EDD; при EDD — User Task запроса доп.документов с петлёй возврата на шаг 4
6. Script Task — формирование оффера: сумма аванса, holdback % (5–25%, в ряде моделей выбирается селлером), общая сумма погашения (repayment cap), эквивалентная % per annum + комиссия за выдачу (обязательно для RU-исполнения; ⚠ Уточнить актуальные диапазоны в продуктовом каталоге), maximum outside date (18–24 мес как watchdog)
7. User Task — селлер акцептует оффер УКЭП ЮЛ/ИП по 63-ФЗ; Timer Boundary (interrupting) 7–30 дней (⚠ Уточнить в каталоге); истечение → End Event «Оффер не акцептован»
8. Sub-process «Pre-disbursement verification» — fresh sanctions pull, PEP-match бенефициаров, сверка реквизитов, повторная проверка статуса селлер-аккаунта (см. будущий процесс; для #16 тот же шаг)
9. Service Task «Disbursement» — перевод на расчётный счёт селлера через расчётный банк, SLA 1–3 рабочих дня; **compensation boundary event** — при выявлении fraud-flag / ошибки KYB / срабатывания санкций после перевода срабатывает compensation handler (reverse-платёж, аннулирование договора, снятие приоритетного удержания); см. compensation pattern в `bpmn-patterns.md`
10. Service Task — активация приоритетного удержания в биллинге маркетплейса; в договорной оферте и в договоре займа должна содержаться оговорка о зачёте встречных однородных требований (ст. 410 ГК РФ) — правовое основание автоматического удержания holdback % из каждой выплаты маркетплейса селлеру
11. Message Start Event (подпроцесс «Holdback cycle») — срабатывает на каждый payout-цикл маркетплейса (в RU-реальности — обычно дважды в месяц, например 15-го и 25-го, либо еженедельно; **НЕ Timer Start Event** как в #10/#16, так как привязка к факту поступления выручки, а не к календарной дате)
12. Script Task — расчёт holdback = min(holdback % × выручка за цикл, remaining cap); если выручка = 0 → удержание = 0
13. XOR gateway — holdback > 0 → Service Task зачёта в биллинге маркетплейса → Service Task уменьшения накопленной суммы задолженности; holdback = 0 → End Event цикла без операций
14. XOR gateway «repayment cap достигнут?» — суммарные удержания сравниваются с cap на каждом цикле; нет → возврат к ожиданию следующего payout-цикла; да → ветка закрытия
15. Timer Boundary (non-interrupting) на родительском процессе — watchdog «нет продаж N дней» (N определяется в договоре, типично 14–30); при срабатывании → escalation sub-process: уведомление селлеру → reassessment → запрос доп. минимального платежа (аналог PayPal 5%/10% каждые 90 дней) → при отказе — переквалификация в досрочное требование → collection sub-process (адаптация процесса 11 для B2B)
16. Timer Boundary (interrupting) на родительском процессе — maximum outside date (18–24 мес); при срабатывании до достижения cap → ветка досрочного требования
17. End Event «RBF закрыт» — при достижении repayment cap; параллельно Service Task снятия приоритетного удержания, Send Task уведомления селлеру и Script Task формирования акта закрытия

**Compliance-аннотации:**
- **RU (кредитор — МКК)**: 151-ФЗ «О микрофинансовой деятельности», ст. 12 ч. 1 п. 8 — предельная сумма микрозайма ЮЛ/ИП **не более 5 млн ₽** (Письмо ЦБ РФ от 05.02.2025 № 44-10/765); ⚠ Уточнить актуальность лимита на дату моделирования
- **RU (кредитор — банк)**: 395-1-ФЗ + Положение ЦБ РФ 590-П по резервам корпоративных ссуд
- **RU правовая конструкция**: **чистый MCA с factor rate** в ГК/151-ФЗ/395-1 не вписывается; рабочая конструкция — «заём + оговорка о зачёте по ст. 410 ГК» (как в #16); альтернативные трактовки — факторинг по ст. 824 ГК (требует уступки будущих требований; ⚠ спорно) или смешанный договор по ст. 421 ГК; в продукте ставка выражается как % per annum + единовременная комиссия, **не как factor rate**
- **RU 115-ФЗ + Положение 499-П** — идентификация ЮЛ/ИП; fresh sanctions pull перед disbursement
- **RU 63-ФЗ** — УКЭП ЮЛ/ИП на акцепте оффера; Госуслуги / ЕСИА для ЮЛ не применяются
- **RU 54-ФЗ** — кассовый учёт при зачёте из выручки: позиции ФНС и методологов расходятся (⚠ Уточнить актуальную позицию ФНС и внутренней бухгалтерии; критично для операционной нагрузки при масштабе маркетплейса)
- **RU 230-ФЗ** — при эскалации через watchdog → ограничения на коммуникацию с должником-ЮЛ; ⚠ Уточнить действие на B2B в текущей редакции
- **RU НК РФ**: ст. 269 (контролируемая задолженность и тонкая капитализация — если кредитор и маркетплейс в одной группе); ст. 251/265 (налоговый учёт процентов у заёмщика); НДС на услуги финансового агента **только при факторинговой квалификации** — ⚠ Уточнить для выбранной конструкции
- **RU 353-ФЗ и ПСК — не применяются** (это норма потребкредита ФЛ)
- **RU Положение ЦБ РФ 4212-У** — отчётность МКК перед ЦБ; сопрягается с процессом 15 Regulatory reporting
- **TR**: **чистый MCA/RBF невозможен без банка-партнёра** — по ст. 13(4) Закона 6493 ödeme kuruluşu kredi verme faaliyetinde bulunamaz; рабочая модель — Servis Bankacılığı по Yönetmelik R.G. 29/12/2021/31704 (маркетплейс = arayüz sağlayıcı, кредитор = лицензированный банк); соответствие Bankacılık Kanunu 5411, BDDK directives; ⚠ Уточнить применимость taksitli ticari kredi / finansman şirketi конструкций
- **KZ**: Law of Kazakhstan «On MFO» — микрокредитование только МФО, ломбардами, кредитными партнёрствами; **AEIR cap 46% годовых** с июня 2025 (ARDFM directive) — effective APR продукта должен вкладываться в cap; ⚠ Уточнить — при holdback > 15% и быстром repayment effective APR легко превышает 46%
- **UZ**: ЗРУ-765 от 20.04.2022 «О небанковских кредитных организациях и микрофинансовой деятельности» — микрокредит субъекту предпринимательства **до 300 млн сум**; принцип срочности обязателен → **наличие maximum outside date в договоре критично**; регулятор — ЦБ Узбекистана
- **ET**: NBE Directive ONPS/10/2025 регулирует PII; специальной digital credit regulation нет — рабочая модель через банк-партнёра (аналог Telebirr↔Dashen Bank); ⚠ Рамка развивается, проверять актуальность перед имплементацией
- **BY**: ⚠ Данных first-party недостаточно; требует отдельного исследования Указов Президента РБ и постановлений НБ РБ по микрофинансовой деятельности
- Срок хранения документов по сделке — минимум 5 лет после прекращения отношений (аналог 115-ФЗ ст. 7 п. 4 в RU; локальные аналоги в других юрисдикциях)

**Особенности моделирования:**
- Collaboration pool обязателен, минимум **4 пула**: **Селлер** (заявка, акцепт, ручное погашение), **Маркетплейс** (витрина, данные продаж для скоринга, биллинг с приоритетным удержанием), **МКК / банк-кредитор** (underwriting, оффер, disbursement, compensation, watchdog, escalation), **Расчётный банк селлера** (зачисление disbursement, при необходимости — списание по акцепту для minimum payment)
- Message flows между пулами, минимум **6**: (1) Селлер → Маркетплейс «заявка» и в обратную «оффер»; (2) Маркетплейс → МКК «поток данных о продажах и возвратах для скоринга и каждого holdback-цикла»; (3) МКК → Расчётный банк «платёжное поручение на disbursement»; (4) Маркетплейс → МКК «сумма зачёта из выручки за payout-цикл»; (5) МКК → Расчётный банк «поручение на прямое списание по акцепту при срабатывании watchdog»; (6) Расчётный банк → МКК «acknowledgement зачисления/списания»
- **Compensation boundary event** на шаге 9 (Disbursement) с атрибутом `isForCompensation="true"` — handler выполняет: reverse-платёж, аннулирование договора, снятие приоритетного удержания; триггер — Compensation Throw Event из ветки fraud-flag или ошибки KYB
- **Continuous debit trigger**: шаг 11 — Message Start Event подпроцесса «Holdback cycle», привязанный к событию payout-цикла маркетплейса (batch-settlement дважды в месяц / еженедельно в зависимости от биллинга), **не Timer Start Event**. Это ключевое отличие от #10/#16: там Timer на плановое списание; здесь триггер — сам факт поступления выручки
- **End condition по cap**: на каждом holdback-цикле (шаг 14) XOR сверка накопленной суммы удержаний с repayment cap; при достижении — срабатывает ветка закрытия, включая снятие приоритетного удержания в биллинге маркетплейса
- **Watchdog**: шаг 15 — Timer Boundary (non-interrupting) «нет продаж N дней» на родительском процессе → escalation sub-process (уведомление → reassessment → запрос минимального платежа → при отказе досрочное требование). Это замена классического collection: «пропуска» планового платежа как такового не существует, реагируем на бездействие
- **Maximum outside date**: шаг 16 — Timer Boundary (interrupting) 18–24 мес; обязателен для выполнения принципа срочности (UZ ЗРУ-765) и для управления concentration risk у кредитора
- Повторные заявки (XOR на шаге 2) идут по сокращённой ветке: пропуск шага 3 KYB и шага 8 части Pre-disbursement verification, ускоренный underwriting с приоритетом внутренних данных маркетплейса, ≤24 часов
- Sub-process'ы, не раскрываемые в этой версии — только ссылки: (1) **Pre-disbursement verification** (как в #16); (2) **RBF escalation** (адаптация процесса 11 для B2B, с учётом 230-ФЗ в адаптированном виде и механики блокировки вывода средств / принудительной распродажи остатков при бездействии)
- Retry-поведение вызовов внешних API и попыток зачёта в биллинге — уровень Camunda runtime (job workers, incident handling, resource allocation), на диаграмме не моделируется
- Именование элементов — object + verb в инфинитиве для задач, object + state для событий, вопрос для XOR-gateway с ответами на исходящих потоках
- Ссылки на переиспользуемые паттерны в `bpmn-patterns.md`: compensation pattern (#8), event-based gateway (#3), approval loop (#1)

# Backend & Compliance

## 14. Settlement / merchant payout

Batch-процесс. Закрывает финансовый цикл с селлерами / партнёрами маркетплейса и с платёжной системой / расчётным банком.

1. Timer Start Event — конец операционного дня (для СБП), конец периода расчёта с селлерами (еженедельно / ежедневно)
2. Агрегация всех транзакций за период — Service Task
3. Расчёт комиссий маркетплейса + налогов — Business Rule Task
4. Формирование netting-позиций по селлерам — Script Task
5. Параллельный шлюз:
   - Выгрузка платёжного поручения в расчётный банк — Send Task
   - Отчёт селлерам (merchant statement) через API / email — Send Task
6. Reconciliation — Receive Task: подтверждение от банка + сверка с partner ledger
7. XOR — расхождения найдены → sub-process dispute resolution с селлером; всё сошлось → End Event «Settlement completed»

**Compliance-аннотации:**
- Платёжные системы: регламенты СБП, Мир, локальных схем по расчётному циклу
- 54-ФЗ (для РФ) — онлайн-касса + чек на каждую транзакцию
- Бухгалтерский учёт: ПБУ 9/99, 10/99 — признание выручки и расходов
- Для маркетплейса в РФ: агентский договор с селлером + чек в пользу принципала

**Особенности моделирования:**
- Reconciliation — почти всегда петля (approval loop): если расхождение, возврат на шаг пересчёта
- Для партнёрских wallet (типа n11 ↔ Fibabanka) — отдельный pool «Банк-партнёр» с message flow для netting reconciliation
- Таймер на Start Event часто cron-подобный: «ежедневно в 02:00 МСК», «еженедельно по понедельникам»

## 15. Regulatory reporting

Batch-процесс формирования и отправки отчётности в регулятор.

1. Timer Start Event — конец отчётного периода (день, декада, месяц, квартал, год)
2. Сбор данных из всех источников — параллельный шлюз с несколькими Service Task (core banking, card processing, wallet, KYC reg, AML alerts)
3. Валидация данных против XBRL / XSD схемы регулятора — Service Task
4. XOR — ошибки валидации → User Task (operator fix) → петля возврата; данные валидны → продолжение
5. Формирование отчётного XML / XBRL — Script Task
6. Подписание УКЭП / локальной e-sign — Service Task
7. Отправка в ЛК регулятора — Send Task (REST / SOAP / file upload)
8. Receive Task — acknowledgment от регулятора
9. XOR — принято / отклонено с ошибками → исправления → повтор; принято → End Event

**Compliance-аннотации:**
- Для РФ: формы ЦБ РФ 0409101, 0409110 (баланс + отчёт о финрезультатах); 0420501 (РСБУ отчётность для НФО); 0409115 для кредитных рисков
- Для UZ: формы ЦБ UZ; для KZ: формы НБК; для BY: формы НБ РБ; для TR: формы BDDK и TCMB
- 115-ФЗ: 0409401 — отчёт об операциях с денежными средствами, подлежащих обязательному контролю
- Сроки: типично 5–25 рабочих дней после отчётного периода; штрафы за просрочку

**Особенности моделирования:**
- Шаг 2 — параллельный fetch из многих источников; обязательно AND gateway
- Ошибки валидации критичны: петля «исправить и перезапустить» (approval loop), лучше с счётчиком итераций — на 3-й ошибке эскалация руководителю
- Aggregate reports (квартальные, годовые) часто требуют свежего closing cycle — может быть блокирован параллельным процессом «Закрытие периода» (sequence или XOR wait)
