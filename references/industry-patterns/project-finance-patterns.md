# Project Finance Patterns — ипотека, застройщики, проектное финансирование

Читается на Step 2, когда domain определён как project finance, construction lending, retail mortgage, SPV financing, infrastructure или workout / problem loans.

Особенность этой отрасли — длительные сроки сделки (месяцы для ипотеки, годы для проектного финансирования), много участников в collaboration, многоуровневое одобрение, обязательный мониторинг после выдачи финансирования.

## 1. Retail mortgage

Массовый потребительский продукт. Отличается от проектного финансирования масштабом, уровнем автоматизации и комплаенс-рамкой (потребительское законодательство, а не корпоративное).

1. Message Start Event — заявка клиента (через Госуслуги / ЛК банка / застройщика)
2. KYC + подтверждение дохода — Service Task: 2-НДФЛ / выписка ПФР / справка работодателя / выписки по счетам для самозанятого
3. Проверка кредитной истории — Service Task (параллельный запрос в НБКИ + ОКБ + Эквифакс)
4. Оценка объекта недвижимости — Service Task: независимый оценщик + интеграция с Росреестром (EGRN выписка для проверки обременений)
5. Расчёт ПДН (показатель долговой нагрузки) — Business Rule Task (DMN)
6. Применение программы господдержки — XOR gateway: семейная / IT-ипотека / льготная / стандартная
7. Решение по заявке — XOR gateway (одобрено / доп.проверка / отказ)
8. Одобрение условий + подписание договора — User Task + e-sign (УКЭП)
9. Государственная регистрация ипотеки — Service Task (Росреестр)
10. Выдача кредита — Service Task (перевод на эскроу-счёт при ДДУ, или напрямую продавцу при вторичке)
11. Мониторинг + страхование — sub-process (ежегодный re-KYC + ежегодная проверка страховки)

**Compliance-аннотации:**
- 102-ФЗ «Об ипотеке» — базовая правовая рамка
- 214-ФЗ — при долевом участии в строительстве (ДДУ), обязательный эскроу-счёт
- 218-ФЗ — государственный кадастр и регистрация прав
- 353-ФЗ ст. 6 — раскрытие ПСК
- ПДН ≤ 50–80% — зависит от программы и категории заёмщика (норматив ЦБ РФ, Указание 5782-У)
- Программы господдержки — отдельные Business Rule Task для каждой: у каждой программы свои критерии (возраст, семейное положение, IT-аккредитация работодателя)
- Срок хранения документов — 5 лет после погашения кредита

**Особенности моделирования:**
- Шаги 2–4 — часто параллельный шлюз (AND split): KYC, кредитная история и оценка идут одновременно для сокращения времени решения
- На шаге 6 XOR по программам — каждая ветка ведёт в отдельный Business Rule Task со своими правилами; по выходу — agregating inclusive gateway (OR join)
- На шаге 9 (регистрация в Росреестре) — Timer Boundary типично 7 рабочих дней; при нарушении — эскалация в бэк-офис
- Шаг 11 (мониторинг) — отдельная диаграмма, запускается как event subprocess с timer в 1 год

## 2. Construction lending (по 214-ФЗ)

Корпоративный продукт. Кредитование застройщика под строительство жилого объекта. Применяется в банках, специализирующихся на жилищном финансировании (АО Банк ДОМ.РФ, Сбер, ВТБ).

1. Message Start Event — заявка застройщика + пакет документов по объекту
2. KYB застройщика + бенефициаров — расширенный (см. `fintech-patterns.md` #2) + проверка опыта реализованных проектов
3. Due Diligence проекта:
   - Юридический — земельный участок, права, градостроительная документация, разрешение на строительство
   - Технический — проектная документация, экспертиза, маркетинговое исследование спроса
   - Финансовый — план доходов (от продаж), план расходов (SMR, инженерия, landscaping), CAPEX schedule
4. Независимая экспертиза проекта (инженер + финансовый аудитор) — Service Task
5. Расчёт рисков и лимита — Business Rule Task (внутренняя модель + требования ЦБ РФ 590-П по резервам на возможные потери)
6. Одобрение с ковенантами — User Task (Credit Committee): темпы продаж (sales velocity), целевое использование средств, порядок залогообеспечения
7. Подписание кредитного договора + договора залога (прав требования по ДДУ) — User Task + e-sign (multi-party)
8. Выдача траншами по этапам строительства (milestone-based disbursement) — Service Task с независимым инженером, проверяющим готовность каждого этапа
9. Мониторинг строительства (ежеквартально) — User Task + Timer boundary: отчёт independent engineer, сверка с планом
10. Погашение за счёт поступлений на эскроу-счета (после завершения объекта) — Service Task

**Compliance-аннотации:**
- 214-ФЗ «Об участии в долевом строительстве» — обязательные эскроу-счета
- 102-ФЗ — залог прав требования
- Положение ЦБ РФ 590-П — требования к резервам по проектному финансированию
- Положение ЦБ РФ 483-П — подходы к оценке кредитного риска при расчёте нормативов
- Обязательный контроль целевого использования — отдельный monitoring sub-process, ежемесячные отчёты застройщика

**Особенности моделирования:**
- Collaboration pool: «Банк» + «Застройщик» + «Independent Engineer» + «Росреестр / Комитет по строительству» + иногда «АО ДОМ.РФ» (при участии госпрограмм)
- Шаг 8 (транши) — циклический процесс: milestone → проверка инженера → разрешение транша → следующий milestone; типично 5–15 итераций за стройку
- Обязательная поглощающая ветка (event subprocess) «ухудшение финансового состояния застройщика» — триггер на enforcement / workout (см. процесс 4)
- Если проект реализуется SPV застройщика (а не самим девелопером) — добавляется процесс 3 (проектное финансирование SPV) вместо этого

## 3. Project finance (SPV, IC, Financial Close)

Крупная инфраструктура, энергетика, недвижимость через отдельную проектную компанию (Special Purpose Vehicle). Самый сложный из процессов в этом файле.

1. Message Start Event — инициация проекта (спонсор / концессионер / застройщик)
2. Создание SPV — Service Task: регистрация юр. лица, capital structuring, opening accounts
3. Комплексный Due Diligence:
   - Технический — feasibility study, engineering review
   - Юридический — permits, licenses, ownership, concession agreement
   - Финансовый — CAPEX, OPEX, revenue forecast, sensitivity analysis
   - ESG — environmental impact assessment, social impact, governance
4. Моделирование cash-flow + sensitivity analysis — Script Task (typical Excel-based)
5. Структура сделки (risk allocation):
   - Offtake agreement (гарантированный покупатель продукта)
   - EPC contract (Engineering, Procurement, Construction с фиксированной ценой и сроком)
   - O&M agreement (Operations & Maintenance на эксплуатационной фазе)
   - Interface agreements между всеми сторонами
6. Multi-level approval — User Task: Credit Committee / Investment Committee / Board (цепочка согласований на разных суммах)
7. Financial Close — parallel gateway: одновременное подписание кредитного договора + equity contribution agreement + guarantees + EPC/O&M/offtake
8. Выдача средств по milestone (construction) — Service Task с независимым инженером; типично 20–40 траншей за период строительства
9. Commissioning + handover — Message Event: завершение строительства, COD (Commercial Operations Date)
10. Operational phase + debt service — sub-process с Timer (ежеквартально debt service, ежемесячно reporting, ежегодный technical audit)
11. Refinancing / экзит — Message End Event (через 5–10 лет после COD): либо рефинансирование, либо продажа проекта, либо full repayment

**Compliance-аннотации:**
- 224-ФЗ «О концессионных соглашениях» (для концессий)
- 115-ФЗ (при международных элементах / cross-border components)
- Положение ЦБ РФ 590-П (резервы)
- Обязательное наличие Independent Engineer (Technical Advisor) и Financial Advisor — закреплено в документации
- При международном участии — санкционный скрининг всех контрагентов и бенефициаров
- ESG standards: IFC Performance Standards (обязательно для DFI-финансирования); национальный ESG-рейтинг

**Особенности моделирования:**
- Обязательно крупный collaboration pool: «Банк-агент» + «Синдикат участников» + «SPV» + «Спонсор(ы)» + «EPC Contractor» + «O&M Operator» + «Offtaker» + «Independent Engineer» + «Insurance» + иногда «DFI / Multilaterals»
- Много message flows между парами — фактически сетевой граф
- Inclusive gateways (OR) для альтернативных источников финансирования: bank loan + bond tranche + equity + subsidies / гранты
- Financial Close — это ОДНОВРЕМЕННОЕ подписание кучи документов; классический parallel gateway (AND split + AND join)
- Event subprocesses для: нарушение ковенантов, material adverse change, force majeure

## 4. Workout

Sub-process для случая, когда проект / кредит становится проблемным. Применяется к ипотеке (retail), застройщикам (corporate), проектному финансированию.

Триггеры: просрочка платежей, нарушение финансовых ковенантов, снижение sales velocity (для застройщиков), technical events (срыв сроков строительства, non-performance EPC).

1. Message Start Event — сработал триггер (из monitoring sub-process или внешнее уведомление)
2. Pre-workout assessment — User Task (специалист по проблемным активам): оценка остаточной стоимости залога, оценка вероятности восстановления, анализ причин
3. XOR — выбор стратегии (ключевой gateway):
   - **Restructure** (реструктуризация) — User Task с заёмщиком: новый график, новые ковенанты, возможно дополнительное обеспечение
   - **Standstill** (мораторий) — Timer-based sub-process: заморозка действий банка на период (3–12 месяцев) в надежде на восстановление
   - **Enforce collateral** (обращение взыскания) — sub-process: уведомление, торги, продажа залога, погашение из вырученных средств
   - **Write-off** (списание) — Service Task: признание убытка; при наличии регресса — передача в collection (см. `fintech-patterns.md` #11)
4. Выбранный путь — execute sub-process
5. End Event по результату: восстановление / частичное возмещение / полный убыток

**Compliance-аннотации:**
- Положение ЦБ РФ 590-П — классификация проблемной задолженности, формирование резервов
- Положение ЦБ РФ 611-П — переуступка прав требования
- ГК РФ ст. 348–350 — обращение взыскания на предмет залога
- Для ипотеки физлиц: 102-ФЗ ст. 50 + 229-ФЗ (исполнительное производство)
- Банкротные процедуры: 127-ФЗ (юр.лица + ИП), 213.1-213.32 (физлица)

**Особенности моделирования:**
- Collaboration: «Банк» + «Заёмщик» + возможно «Внешний управляющий / Арбитражный» + «Суд» + «ФССП» + «Независимый оценщик»
- На шаге 3 XOR — редко one-shot решение; часто сначала пробуют standstill, потом restructure, и только потом enforcement. Моделируется через последовательный approval loop с параметром «attempts»
- Обязательная аннотация про классификацию задолженности по 590-П — от этого зависит размер резерва (от 1% для standard до 100% для loss)
